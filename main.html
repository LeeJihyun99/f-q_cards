My experience
University team project of implementing board game with Java

In my 5th semester, I worked on a team project where we had to implement an existing board game called 'Robo Rally.' We used:

  • JavaFX and SceneBuilder for the UI/UX
  • Java for the game logic

Since I was particularly interested in frontend development, I took on the responsibility of UI/UX design and its implementation. My primary focus was to ensure that the design was intuitive, so players could easily follow the game without confusion.

1. Creating a Waiting Room
  • One of my tasks was to create a waiting room where players would wait until all were ready.
  • I remembered a game I used to play where I got bored and confused waiting for it to start because the signals were unclear or not noticeable, even after updates.
  • To avoid this problem, I added a crown icon for the player who created the room so that everyone would know who the leader was.
  • The room creator (leader) could choose a map and wait for everyone to be ready.
  • I made it clear to all players when the leader was choosing the map, and I added a toggle button for each player to indicate whether they were ready or not.
2. Handling Complex Player Actions
  • During the game, each player takes turns selecting multiple cards that determine their robot's movements for the next round.
  • Given the complexity of actions, I anticipated that players might get confused.
  • To address this, I designed a small space on the side of the screen displaying the status of all players, including the cards they chose, represented with small images for clarity.
3. Drag-and-Drop Functionality
  • To enhance visual understanding, I added a drag-and-drop feature at the bottom of each player's screen.
  • Instead of just showing the card names, I used images of the cards, allowing players to easily drag and drop the cards they wanted to use for the next round.
4. Additional Card Selection
  • In the middle of the game, players could choose additional cards.
  • Instead of replacing the cards they had already chosen at the bottom of the screen, I created a popup box for selecting additional cards.
  • This allowed players to view both the cards they had already chosen and the additional cards they needed to select without confusion.
Focus on UI Clarity

Throughout the project, I ensured that each function had its own clear and distinct space within the UI. This was essential for creating an interface that was easy to navigate and prevented confusion during gameplay.

Working student at AdessoSE

I worked at Adesso SE for a year, where I participated in multiple projects, primarily using React.js and occasionally Vue.js. Some of these were large internal projects that were already ongoing, and my task was to implement new features.

Key Projects and Responsibilities 1. Login Page Implementation

One of the first tasks I took on was implementing the login page using React.js. This was my first experience working with React, and it served as a great learning opportunity to understand how to build UI components efficiently.

2. Filter Page

Another major task was building a filter page where users could filter items based on specific features. The page would dynamically show the filtered results, depending on the user's input.

3. Keycloak Integration Issue

During one project, we encountered significant issues while integrating Keycloak with the backend. Keycloak is an open-source identity and access management solution that provides features like SSO (Single Sign-On) and identity brokering. It connects the frontend and backend by generating a token (JWT) that the backend validates to control user authentication and authorization.

In our project, there was a misunderstanding between the backend and frontend teams regarding how Keycloak should be used. The frontend was initially attempting to handle more of the Keycloak process than necessary. After becoming the only frontend developer on this project, I had to communicate with the backend team and the product manager to clarify how Keycloak should work in this flow.

After gathering the necessary information, I created a diagram to outline how the flow should function:

  • Keycloak should be handled entirely on the backend.
  • The backend would provide a token, and the frontend would simply use that token for authentication.

I then adjusted the incorrectly implemented frontend logic, ensuring that it was aligned with the clarified process. This experience taught me the importance of clear communication and detailed documentation to avoid wasting time on implementation misunderstandings.

Lessons Learned

In smaller projects, where I only needed to implement frontend features such as filters or forms, the process went much smoother. These projects were more successful because:

  • We had regular communication through daily meetings.
  • If someone couldn not attend, we would leave questions or updates in our group chat, which we checked daily.

In one such project, I worked with two other team members, and thanks to the clear communication, we were able to finish everything on time and successfully present it to the project head.
